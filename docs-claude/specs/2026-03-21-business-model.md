## 1. 비즈니스 모델 (BM)

### 1.1 시장 분석

#### 한국 경조사 시장 규모

- 한국 가구당 연평균 경조사비 지출: 약 120~180만 원 (통계청 기준)
- 결혼식 1건 평균 축의금 수수 건수: 100~300건
- 장례식 1건 평균 조의금 수수 건수: 50~200건
- 경조사비 관리 니즈 보유 인구: 경제활동인구 약 2,800만 명 중 30~60대 핵심 타겟 약 1,500만 명
- 디지털 경조사 관리 시장 침투율: 약 5% 미만 (대부분 수기/엑셀 관리)

#### 경쟁사 분석

| 서비스 | 주요 기능 | 수익 모델 | 강점 | 약점 |
|--------|----------|----------|------|------|
| 부조장부 (앱) | 기본 기록, 간단 통계 | 광고 + 프리미엄 | 간단한 UI, 높은 인지도 | 기능 제한, 분석 부재 |
| 축의금 관리 (앱) | 결혼 특화 기록 | 광고 | 결혼 특화 | 경조사 전반 미지원 |
| 경조사 가계부 (앱) | 수입/지출 기록 | 무료 (광고) | 무료 | 업데이트 중단, 낮은 UX |
| 네이버 가계부 | 범용 가계부 | 플랫폼 연계 | 브랜드 신뢰, 대규모 사용자 | 경조사 특화 아님 |
| 엑셀/수기 | - | - | 자유도 높음 | 관리 불편, 분석 불가 |

#### MEMON의 차별적 경쟁 우위 (Competitive Moat)

1. **스마트 금액 추천 엔진**: 개인+집단 데이터 기반 3단계 추천 (경쟁사 없음)
2. **물가환산 계산기**: CPI/예금이자/커스텀 이자율 기반 과거 금액 현재가치 환산 (유일)
3. **관계 타임라인 + 물가보정**: 관계별 거래 히스토리를 물가보정하여 시각화 (유일)
4. **트렌드 인사이트**: 집단 통계 기반 나의 경조사비 위치 파악 (유일)
5. **경조사 예산 플래너**: 월별/연간 예산 설정 + 예정 지출 연동 (유일)
6. **관계연결**: 다른 사용자와 데이터 공유/연동 (유일)
7. **다크 모드, 전역 검색, CSV 내보내기, 일괄 등록** 등 프로덕션 수준 UX

## 2. 결제 전략

### 2.1 결제 모델 비교

#### 구독 모델 (Subscription) vs 인앱 결제 (IAP)

| 항목 | 구독 모델 | 인앱 결제 (일회성) |
|------|---------|----------------|
| 수익 예측 가능성 | 높음 (MRR/ARR 산정 가능) | 낮음 (불규칙) |
| LTV (고객 생애 가치) | 높음 (12개월+ 유지 시) | 낮음 (1회 결제) |
| 사용자 부담감 | 중간 (매월 과금 인지) | 낮음 (1회 결제 후 잊음) |
| 이탈율 (Churn) | 관리 필요 (월 5~10% 예상) | 해당 없음 |
| 앱스토어 수수료 | 15~30% (구독은 2년차부터 15%) | 15~30% |
| 기능 업데이트 동기 | 높음 (지속적 가치 제공 필수) | 낮음 |

**권장: 구독 모델 (월/연)**

MEMON의 핵심 가치는 "지속적 데이터 축적"에 있으므로, 구독 모델이 서비스 특성에 부합한다. 데이터가 쌓일수록 추천 정확도가 높아지고, 통계 분석이 풍부해지므로 구독 해지 장벽(Switching Cost)이 자연스럽게 형성된다.

## 4. Web 광고 배치 분석

### 4.1 핵심 전략: 반응형 사이드 광고 + Content max-width

Content 영역에 `max-width`를 지정하고, 브라우저가 이보다 넓을 때 좌우 여백에 광고를 배치한다.
좁을 때는 Content 상단 배너로 폴백한다. 이 방식은 Content 영역을 침범하지 않으면서 넓은 화면을 수익화한다.

### 4.2 레이아웃 구조

**현재:**
```
+---Sidebar(200px)---+---Content(flex-1, 무제한)---+
```

**변경 후:**
```
+---Sidebar(200px)---+--AD--+--Content(max-1280px)--+--AD--+
                     160px   mx-auto 중앙 정렬       160px
```

**Content max-width: `1280px`**
- 현재 모든 페이지 콘텐츠가 1280px 이내에서 충분히 표시됨
- Popup L 사이즈가 1200px → 1280px이면 모든 팝업 포함
- Sidebar 200px + Content 1280px + Side Ads 160px×2 = **1800px** 이상 화면에서 양쪽 광고 노출

### 4.3 반응형 동작

| 브라우저 너비 (Sidebar 제외) | 동작 |
|--------------------------|------|
| **> 1280px** (초과) | Content 중앙 정렬 + **좌우 사이드 광고** (160×600 Skyscraper) |
| **≤ 1280px** (이하) | Content 100% + **상단 배너 광고** (728×90 Leaderboard) |
| **< 768px** (모바일) | Content 100% + **상단 배너 광고** (320×100 Mobile Banner) |

### 4.4 시각적 레이아웃

#### Wide Screen (> 1280px 여유 공간)

```
+---Sidebar---+------Main Area (flex-1)------+
|             |                              |

|             |  |160x  | max-1280 |160x  | |
|  📊 대시보드 |  |600   | mx-auto  |600   | |
|  👥 관계관리 |  |      |  p-24    |      | |
|  🔗 관계연결 |  |Wide  |          |Wide  | |
```

### 4.5 광고 슬롯 사양

| 슬롯 | 광고 크기 | 조건 | 예상 CTR | 가시성 |
|------|---------|------|---------|--------|
| **Side Left** | 160×600 Wide Skyscraper | Content 외부 여백 > 160px | 0.8~1.5% | 높음 (상시 노출) |
| **Side Right** | 160×600 Wide Skyscraper | Content 외부 여백 > 160px | 0.8~1.5% | 높음 (상시 노출) |
| **Top Banner** | 728×90 Leaderboard (반응형) | 사이드 광고 미표시 시 | 0.5~1.2% | 높음 |

**핵심 원칙:** 사이드 광고와 상단 배너는 **동시에 표시하지 않음**. 화면이 넓으면 사이드, 좁으면 상단.

### 4.6 구현 설계

#### AdminLayout 구조 변경

```
AdminLayout
├── Sidebar
└── <main> (flex-1)
    ├── Header
    └── <div> (flex-1, overflow-y-auto)
        └── AdLayout (신규 wrapper)
            ├── [Side Ad Left]   ← position: sticky, top: 0
            ├── <div> (max-w-1280px, mx-auto, w-full, p-24)
            │   ├── [Top Banner]  ← 사이드 광고 미표시 시만
            │   └── {children}    ← 기존 페이지 콘텐츠
            └── [Side Ad Right]  ← position: sticky, top: 0
```

#### AdLayout 컴포넌트 설계

```typescript
// shared/components/AdLayout.tsx

interface AdLayoutProps {
  children: ReactNode
}

export function AdLayout({ children }: AdLayoutProps) {
  const { subscription } = useUserSubscription()
  const isFree = subscription?.tier === 'FREE' || !subscription

  // Premium/Pro 사용자는 광고 없이 기존과 동일하게 렌더링
  if (!isFree) {
    return (
      <div className="max-w-[1280px] mx-auto w-full p-24">
        {children}
      </div>
    )
  }

  return (
    <div className="flex justify-center w-full">
      {/* Left Side Ad — 여백이 있을 때만 표시 */}
      <div className="hidden 2xl:block shrink-0 w-[160px] sticky top-0 self-start pt-24">
        <AdBanner slot="side-left" />
      </div>

      {/* Content */}
      <div className="max-w-[1280px] w-full p-24">
        {/* Top Banner — 사이드 광고가 없을 때만 표시 */}
        <div className="block 2xl:hidden mb-16">
          <AdBanner slot="top-banner" />
        </div>
        {children}
      </div>

      {/* Right Side Ad — 여백이 있을 때만 표시 */}
      <div className="hidden 2xl:block shrink-0 w-[160px] sticky top-0 self-start pt-24">
        <AdBanner slot="side-right" />
      </div>
    </div>
  )
}
```

#### Tailwind Breakpoint 활용

| Breakpoint | 조건 | 동작 |
|-----------|------|------|
| `2xl` (1536px) | Main 영역 ≥ 1536px (= 1280 + 160×2 여유) | 사이드 광고 표시 |
| `< 2xl` | Main 영역 < 1536px | 상단 배너로 폴백 |
| `< md` (768px) | 모바일 | 320×100 모바일 배너 |

> **참고:** Sidebar 200px이 별도이므로, 전체 화면 너비 1736px 이상에서 사이드 광고가 표시됨.
> 일반적인 27인치 모니터(2560px), 울트라와이드(3440px)에서 최적.

### 4.8 광고 UX 가이드라인

| 규칙 | 설명 |
|------|------|
| **사이드 광고 = Content 비침범** | 사이드 광고는 Content max-width 바깥에만 배치, 콘텐츠 축소 없음 |
| **상단 배너 = 최소 높이** | 728×90px, Content 상단에 배치하되 mb-16으로 간격 확보 |
| **기록 입력 중 광고 유지** | 사이드 광고는 Content를 침범하지 않으므로 폼 작성 시에도 유지 가능 |
| **모달/드로어 위 광고 금지** | 오버레이 UI 위에 광고 겹침 금지 |
| **로딩 중 광고 허용** | 사이드 광고는 콘텐츠와 독립적이므로 즉시 렌더링 가능 |
