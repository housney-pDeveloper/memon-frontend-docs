# 배포팀 (DevOps Team)

## 팀 미션

품질관리를 통과한 코드의 빌드 검증, 배포, 릴리스 버전 관리를 수행합니다.

## 담당 워크플로우 단계

**빌드, 배포**

## 팀 구성

### 필수 에이전트

| 에이전트 | 파일 | 역할 |
|----------|------|------|
| **빌드/배포 에이전트** | `agent/07_build-deployer.md` | 빌드 검증, 테스트 실행, 환경별 배포, 서브모듈 관리 |

### 보조 에이전트

| 에이전트 | 파일 | 역할 |
|----------|------|------|
| **릴리스 관리 보조** | `agent/sub_release-manager.md` | 버전 관리(SemVer), 변경 로그, 배포 후 스모크 테스트, 롤백 절차 |

## 입력 ← 품질관리팀 통과 보고서

- 코드 리뷰 결과 (통과)
- 보안 검토 결과 (통과)
- 접근성/성능 검토 결과 (참고)
- 배포 승인 상태

## 서브모듈 커밋 순서 (중요)

```
hs-common 변경 있는 경우:
  1. cd 21_hs-common → commit → push
  2. cd 31_webadmin → 빌드 확인 → commit → push
  3. cd 32_appuser → 빌드 확인 → commit → push
  4. cd workspace → 서브모듈 해시 반영 → commit → push

개별 프로젝트만 변경된 경우:
  1. cd {project} → commit → push
  2. cd workspace → 서브모듈 해시 반영 → commit → push
```

## 커밋 메시지 규칙

GIT_POLICY.md 준수:

```
type

body (한 줄 72자 이내, 한글)

footer (유형: #이슈번호)
```

- type: feat / fix / docs / style / refactor / test / chore
- Subject: 50자 이하, 한글, 동사 원형 + 대문자 시작

## 환경별 배포 설정

| 환경 | 빌드 명령어 | 환경 변수 | 브랜치 |
|------|------------|-----------|--------|
| 로컬 개발 | `npm run dev` | `.env` | feature/* |
| 개발 서버 | `npm run build` | `.env.dev` | develop |
| 운영 서버 | `npm run build:prod` | `.env.prod` | main |

## 협업 규칙

1. **사전 검증 실패 시 개발팀에 반환** — 빌드 진행하지 않음
2. **서브모듈 커밋 순서 엄격 준수** — hs-common → 개별 프로젝트 → 워크스페이스
3. **운영 배포 전 개발 환경에서 먼저 검증** — 개발 빌드 성공 확인 후 운영 빌드
4. **릴리스 관리 보조가 버전과 변경 로그를 관리** — 빌드/배포 에이전트는 실행에 집중
5. **배포 실패 시 즉시 롤백 판단** — P0/P1은 즉시 롤백, 그 외는 원인 분석 후 결정
6. **배포 후 스모크 테스트 필수** — 주요 기능 정상 동작 확인 후 배포 완료 선언
