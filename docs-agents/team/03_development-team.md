## 입력 ← 설계팀 산출물

- 디렉토리 구조 및 파일 목록
- 컴포넌트 설계서 (Props, 상태 의존성)
- API 인터페이스 (타입, 엔드포인트)
- 상태 관리 설계 (쿼리 키, 스토어)
- 구현 순서 및 주의 사항

## 개발 순서 원칙

### 의존성 기반 순서

```
1. hs-common (공통 타입/유틸리티)  ← 항상 최우선
2. publishing (마크업 참조)        ← webadmin/appuser와 병렬 가능
3. webadmin + appuser             ← 병렬 개발
4. 테스트 작성                     ← 구현 완료 후
```

### 프로젝트 내부 순서

```
1. 타입 정의 (types.ts)
2. API 함수 (entities/api → features/api)
3. 쿼리/뮤테이션 훅 (model/)
4. UI 컴포넌트 (entities/ui → features/ui)
5. 페이지 구성 (app/ 라우트)
6. 가드/권한 적용
7. 메뉴/네비게이션 등록
```

## 협업 규칙

1. **hs-common 변경은 항상 최우선** — 양쪽 프로젝트 빌드 확인 후 다음 단계 진입
2. **webadmin과 appuser는 병렬 개발** 가능하나, 공통 부분은 hs-common에서 먼저 해결
3. **퍼블리싱은 UI 참조 목적** — 비즈니스 로직 포함 금지, 개발 에이전트가 참고용으로 활용
4. **테스트 엔지니어는 구현 완료 후 투입** — 구현 중 변경이 잦으므로 완료 후 작성
5. **서브모듈 커밋 순서 준수**: hs-common → 개별 프로젝트 → 워크스페이스 해시 반영
6. **설계 대비 변경 발생 시 설계팀에 피드백** — 구현 메모에 기록하고 설계 업데이트 요청
